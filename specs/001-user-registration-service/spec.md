# Feature Specification: 使用者註冊與電子郵件驗證

**Feature Branch**: `001-user-registration-service`  
**Created**: 2026-03-01  
**Status**: Draft  
**Input**: User description: "建立一個使用者註冊服務，包含電子郵件驗證和密碼加密。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 建立帳號並完成驗證 (Priority: P1)

作為新使用者，我可以提交電子郵件與密碼建立帳號，並透過收到的驗證郵件完成電子郵件驗證，確保帳號可安全啟用。

**Why this priority**: 這是整個註冊流程的核心價值，若無法註冊與驗證，功能無法交付。

**Independent Test**: 可獨立以「提交註冊表單 → 收到驗證郵件 → 點擊驗證連結 → 帳號狀態變為已驗證」完整測試，且能直接產生可用帳號。

**Acceptance Scenarios**:

1. **Given** 訪客尚未註冊且輸入有效電子郵件與符合規則的密碼，**When** 送出註冊，**Then** 系統建立未驗證帳號並發送驗證郵件。
2. **Given** 使用者持有有效且未過期的驗證連結，**When** 點擊連結，**Then** 系統將帳號標記為已驗證並回覆成功訊息。

---

### User Story 2 - 防止未驗證帳號直接使用服務 (Priority: P2)

作為系統管理方，我需要未完成電子郵件驗證的帳號無法進入受保護功能，以降低假帳號與濫用風險。

**Why this priority**: 驗證狀態控管直接影響帳號安全與服務品質，屬於核心安全防線。

**Independent Test**: 可獨立建立未驗證帳號並嘗試使用需登入/已啟用權限功能，驗證其被阻擋且收到明確提示。

**Acceptance Scenarios**:

1. **Given** 帳號尚未驗證，**When** 嘗試使用僅限已驗證帳號的功能，**Then** 系統拒絕並提示需先完成電子郵件驗證。
2. **Given** 帳號已完成驗證，**When** 使用相同受保護功能，**Then** 系統允許通行。

---

### User Story 3 - 密碼安全保存與重複註冊處理 (Priority: P3)

作為產品負責人，我需要系統以安全方式儲存密碼，並對重複電子郵件註冊提供一致回應，避免帳號安全與使用者體驗問題。

**Why this priority**: 屬於安全與資料一致性要求，對長期維運與風險控管重要。

**Independent Test**: 可獨立檢查資料儲存內容不含明文密碼，並驗證同一電子郵件重複註冊時回應符合規格。

**Acceptance Scenarios**:

1. **Given** 使用者完成註冊，**When** 檢視使用者密碼儲存資料，**Then** 不可出現可逆的明文密碼內容。
2. **Given** 系統已存在相同電子郵件帳號，**When** 再次提交註冊，**Then** 系統回覆一致且可理解的結果，且不建立重複帳號。

### Edge Cases

- 驗證連結已過期或已使用時，系統應拒絕驗證並提供可再次發送驗證郵件的引導。
- 使用者輸入格式錯誤的電子郵件或不符合密碼規則時，系統應即時拒絕並明確說明原因。
- 驗證郵件暫時無法送達時，系統應回報可理解訊息並保留可重試流程。
- 使用者在短時間重複送出註冊時，系統應避免建立多筆重複資料。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 提供使用者註冊流程，接受電子郵件與密碼建立新帳號。
- **FR-002**: 系統 MUST 在建立帳號後將其預設為「未驗證」狀態，直到完成電子郵件驗證。
- **FR-003**: 系統 MUST 發送含一次性驗證連結的驗證郵件給新註冊使用者。
- **FR-004**: 系統 MUST 驗證驗證連結的有效性（包含存在性、未過期、未被使用），並在成功後將帳號標記為「已驗證」。
- **FR-005**: 系統 MUST 以不可還原為明文的方式儲存使用者密碼。
- **FR-006**: 系統 MUST 防止相同電子郵件建立重複帳號。
- **FR-007**: 系統 MUST 在未驗證帳號嘗試存取受保護功能時拒絕請求並回傳明確提示。
- **FR-008**: 系統 MUST 提供重新發送驗證郵件的能力給尚未驗證的帳號。
- **FR-009**: 系統 MUST 記錄註冊、驗證成功、驗證失敗與重發驗證信等關鍵事件，供稽核與問題追蹤。

### Non-Functional Requirements *(mandatory)*

- **NFR-QUALITY-001**: 程式碼變更 MUST 通過專案既有的 lint、format 與型別檢查。
- **NFR-TEST-001**: 每一項功能需求 MUST 對應可執行且可驗證的測試案例。
- **NFR-UX-001**: 使用者在註冊與驗證流程中 MUST 取得一致、明確且可行動的回饋訊息。
- **NFR-PERF-001**: 一般負載下，註冊請求從送出到收到處理結果應在 2 秒內完成（不含郵件實際到達時間）。
- **NFR-PERF-002**: 電子郵件驗證結果頁面在有效連結情境下應於 2 秒內回覆成功或失敗結果。
- **NFR-SEC-001**: 註冊與驗證相關敏感資料（如密碼、驗證 token）不得以明文形式出現在持久化資料或一般操作日誌中。

### Key Entities *(include if feature involves data)*

- **User Account**: 代表一個使用者身份，包含電子郵件、密碼雜湊值、帳號建立時間、驗證狀態與驗證完成時間。
- **Email Verification Token**: 代表一次性電子郵件驗證憑證，包含關聯帳號、到期時間、使用狀態與發送時間。
- **Registration Audit Event**: 代表註冊與驗證流程中的關鍵操作紀錄，包含事件類型、觸發時間、帳號識別資訊與結果。

## Assumptions

- 本功能以一般最終使用者為主要對象，不含企業多租戶或管理員批次建立帳號流程。
- 驗證郵件可在短時間內送達；若外部郵件服務延遲，系統仍需提供重發機制。
- 密碼規則至少包含最小長度與基本強度要求，具體規則由產品既有政策沿用。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 90% 以上新使用者可在 3 分鐘內完成「註冊提交至電子郵件驗證完成」流程。
- **SC-002**: 在有效資料輸入情境下，95% 註冊請求可於 2 秒內回覆可理解結果。
- **SC-003**: 上線後首月，因「未收到驗證信或驗證失敗」產生的客服工單比例低於新註冊量的 5%。
- **SC-004**: 100% 新建立帳號皆以未驗證狀態起始，且未驗證帳號對受保護功能的存取阻擋率為 100%。
- **SC-005**: 密碼資料稽核抽樣中，0 筆資料以明文形式儲存。
